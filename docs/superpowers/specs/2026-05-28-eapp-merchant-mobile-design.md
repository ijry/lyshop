# 商家后台移动端 eapp 设计（最新架构）

## 1. 背景与目标

为单店铺商城新增独立商家移动端工程 `eapp/`，覆盖 `H5 + 微信小程序 + App` 三端，承接现有后台管理能力，形成移动化运营闭环。

本次目标：

1. 新建独立工程，不与现有用户端 `app/` 混用路由与状态。
2. 商家端完整模块可用：工作台、订单、商品、营销、消息、店铺、员工与权限。
3. 接口层复用现有后台接口语义，不新增商家前缀。
4. 采用“行业常规商家后台结构 + 任务优先视觉节奏”的移动商家体验。

---

## 2. 约束与边界

1. 工程形态：必须为独立工程 `eapp/`。
2. 能力范围：按“完整版”交付，不做仅 MVP 裁剪。
3. 端能力：统一支持 `H5 + 微信小程序 + App`。
4. 接口策略：直接复用现有后台接口，不额外新增 `/merchant/*` 前缀。
5. 业务定位：单店铺场景，现有后台即商家后台。

---

## 3. 方案对比与决策

### 方案 A：全原生模块化（选型）

- 方式：`eapp` 全量页面使用 `uni-app + uview-plus` 原生实现，模块化拆分。
- 优点：三端一致性好、交互可控、长期维护成本低。
- 成本：首期开发面较大。

### 方案 B：原生壳 + WebView 混合

- 方式：高频页原生，低频页嵌入后台 H5。
- 优点：可快速上线。
- 缺点：体验割裂、登录态同步复杂、技术债高。

### 方案 C：小程序优先再补齐

- 方式：优先满足小程序，再扩展 H5/App 差异。
- 优点：分期风险可控。
- 缺点：首期跨端一致性不足。

结论：采用方案 A，并按里程碑拆分实施任务。

## 4. 总体架构设计

### 4.1 工程结构

新增目录 `eapp/`，建议结构：

```text
eapp/
├── pages/                 # 业务页面（按模块分组）
├── pages-sub/             # 低频/深层页面分包
├── api/                   # 模块接口封装
├── stores/                # Pinia 状态（auth/app/badge）
├── components/            # 复用业务组件
├── utils/                 # request、permission、platform 等
├── locales/               # 多语言（与现有仓库风格一致）
├── static/                # 品牌与业务静态资源
├── pages.json
├── manifest.json
└── main.ts
```

### 4.2 技术栈

1. `uni-app` + `Vue3` + `TypeScript`
2. `uview-plus` 作为主组件库
3. 与现有仓库一致的国际化与状态管理模式（`vue-i18n + pinia`）

### 4.3 登录态与权限

1. 登录接口复用后台登录能力。
2. Token 语义与后台一致，移动端本地存储键采用 `eapp_admin_token`（避免与 `admin` 本地缓存冲突）。
3. 权限判断沿用后台权限码语义，支持页面级与操作级控制。

## 5. 信息架构与导航设计

### 5.1 底部主导航（5 Tab）

1. 工作台
2. 订单
3. 商品
4. 营销
5. 我的

### 5.2 二级导航组织

1. 工作台：经营概览、待办事项、快捷入口
2. 订单：订单列表、订单详情、售后处理、发货操作
3. 商品：商品列表、上下架、库存与价格快捷改、商品编辑
4. 营销：优惠券、秒杀、拼团、砍价入口与管理
5. 我的：消息中心、IM 会话、店铺设置、管理员、角色、退出登录

### 5.3 导航原则

1. 一级入口稳定且高频，避免桌面后台深菜单平移到移动端。
2. 高频任务两步可达，减少跨页跳转深度。
3. 待办卡片与业务页筛选参数联动跳转。

## 6. UI 设计系统（ui-ux-pro-max 落地）

本项目采用“结构稳 + 视觉高识别”的移动商家风格，不使用高成本重玻璃效果。

### 6.1 设计令牌

1. `primary`: `#2563EB`
2. `success`: `#16A34A`
3. `warning`: `#F59E0B`
4. `danger`: `#DC2626`
5. `accent`: `#F97316`
6. `bg`: `#F8FAFC`
7. `card`: `#FFFFFF`
8. `text`: `#1E293B`

### 6.2 字体策略

1. 主体：`HarmonyOS Sans SC / Alibaba PuHuiTi 3.0 / sans-serif`
2. 数字：`Rubik / DIN Alternate / sans-serif`

### 6.3 组件风格

1. 卡片圆角 `12-16`，轻阴影，状态色标识条。
2. 主按钮实色、次按钮描边，强调任务层级。
3. 列表优先卡片化，避免移动端宽表格。

### 6.4 交互与可用性

1. 动效时长控制在 `150-220ms`。
2. 触控目标最小 `44px`。
3. 标题层级连续，不跳级。
4. 返回行为符合端侧预期，不破坏历史栈。
5. 输入框按类型调起对应键盘。

## 7. 接口复用与数据流设计

### 7.1 接口复用原则

1. `baseURL` 统一为 `/admin/api`。
2. 复用现有后台接口语义，不新增商家接口前缀。
3. 请求统一封装 `code/msg/data` 处理与错误抛出。

### 7.2 模块接口映射

1. 鉴权：`/auth/login`、`/menus`
2. 工作台：后台 dashboard 统计接口
3. 订单：订单列表/详情/发货/售后相关接口
4. 商品：商品列表、商品编辑、上下架、库存价格更新接口
5. 营销：优惠券、秒杀、拼团、砍价接口
6. 消息与会话：message、im 相关接口
7. 店铺与权限：site settings、admins、roles 接口

### 7.3 状态流

1. 页面筛选参数统一进入 store，支持回退保留与列表恢复。
2. 关键动作（发货、上下架、发送消息）完成后局部刷新，并同步更新工作台待办计数。
3. Tab 徽标由统一 badge store 驱动，来源于订单、售后、消息聚合统计。

## 9. 错误处理与安全控制

1. `401`：清理本地登录态并跳转登录。
2. `403`：提示权限不足并保留当前上下文。
3. 网络异常：统一错误提示 + 可重试入口。
4. 空态：各模块提供业务化空态文案与引导操作。
5. 幂等：关键提交按钮防重复点击，避免重复提交。

## 10. 测试与验收标准

### 10.1 测试层次

1. 单元测试：请求封装、权限判断、状态映射。
2. 页面联调：登录、发货、商品上下架、营销编辑、消息发送。
3. 多端回归：H5/微信小程序/App 关键链路一致性。

### 10.2 验收标准

1. 五个 Tab 主流程全部可闭环。
2. 订单/商品/营销核心动作与后台语义一致。
3. 无权限功能不可误操作。
4. 关键页面首屏可用，无明显卡顿或布局错位。
