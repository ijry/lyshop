# docs-site 新增「AI花絮」导航与内容整理设计

## 1. 背景与目标

当前 `docs-site` 已有 AI 相关内容分散在 `guide/features.md`、`README.md` 等文档中，但缺少统一入口和结构化阅读路径。

本次目标：

1. 在顶部主导航新增「AI花絮」入口（放在导航最后一项）。
2. 采用“索引页 + 子页分拆”方式，整理现有 AI 内容。
3. 明确内容来源和边界：仅整理现有文档，不新增未落地事实。

## 2. 范围

### 2.1 In Scope

- 更新 `docs-site/docs/.vitepress/config.mts` 顶部导航与 AI 专区侧边栏。
- 新增 AI 花絮文档目录与 4 个页面：
  - `docs/ai-notes/index.md`
  - `docs/ai-notes/capabilities.md`
  - `docs/ai-notes/api.md`
  - `docs/ai-notes/evolution.md`
- 从既有文档提炼并重组内容。

### 2.2 Out of Scope

- 新增后端 AI 接口或修改服务端实现。
- 编造当前文档未覆盖的接口字段。
- 修改非 docs-site 文档站点结构。

## 3. 信息架构

### 3.1 顶部导航

- 在 `themeConfig.nav` 最后一项新增：`AI花絮` → `/ai-notes/`。

### 3.2 目录结构

- `ai-notes/index.md`：导航门页，说明阅读路径与页间关系。
- `ai-notes/capabilities.md`：能力总览（模型支持、业务入口、工作流定位）。
- `ai-notes/api.md`：接口速查（仅基于 docs-site 现有公开信息）。
- `ai-notes/evolution.md`：演进记录与实践要点。

### 3.3 专区侧边栏

- 为 `/ai-notes/` 增加侧边栏分组 `AI花絮`，包含上述 4 页，便于分章节阅读。

## 4. 内容来源映射

### 4.1 capabilities.md

来源：

- `docs-site/docs/guide/features.md` 中 AI 生图章节。
- `README.md` 中 AI 生图与工作流描述。

整理后覆盖：

- AI 生图能力边界。
- 支持模型范围。
- 在商品编辑页中的集成定位。
- 使用链路（配置模型 → 生成图片 → 应用到商品素材）。

### 4.2 api.md

来源：

- `docs-site` 当前已公开的 API 文档可见内容（优先 `api/*`）。
- 若 docs-site 中尚未细化 AI 专属接口文档，则保留“当前文档覆盖范围说明”。

整理后覆盖：

- 已公开的接口入口或能力说明。
- 与 AI 使用链路相关的请求/响应关键点（仅引用现有信息）。
- 文档一致性提示：以 API 文档实际章节为准。

### 4.3 evolution.md

来源：

- `docs-site/docs/guide/features.md` 中“AI 生图已内嵌商品编辑页”相关演进描述。

整理后覆盖：

- 从独立入口到业务内嵌的演进逻辑。
- 对后台信息架构与运营效率的影响。
- 文档层实践建议（如何选模型、如何组织素材、如何避免误配）。

## 5. 文案与事实约束

- 不新增未验证接口字段。
- 不引入 docs-site 之外未发布结论作为“已上线事实”。
- 对“待补完”的能力以“当前文档未覆盖”表达，不使用含糊占位词。

## 6. 验收标准

1. 顶部导航最后一项可见 `AI花絮`，可跳转到 `/ai-notes/`。
2. `AI花絮` 专区四个页面可互相切换，侧边栏路径正确。
3. 内容均可在已有文档中追溯来源，无事实冲突。
4. `npm run docs:build` 在本地可通过。

## 7. 部署与配置影响

- 无后端配置变更。
- 仅 docs-site 文档结构与导航更新。
- 构建流程沿用现有 `docs:build`（已接入预清理脚本）。

## 8. 决策记录

- 决策 1：采用“索引页 + 子页分拆”而非单页聚合。
- 决策 2：`AI花絮` 放在顶部导航最后一项。
- 决策 3：仅整理现有文档内容，不扩展未落地事实。
